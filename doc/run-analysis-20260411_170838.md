## 2. 一个非常关键的数字：`61` 次请求几乎正好说明没有工具调用

这次运行总共有 `61` 次 LLM 请求。这个数字非常有信息量。

在 ax-prover 的 iterative 模式中，若每轮都发生编译失败，则通常会经历：

1. proposer 调一次模型
2. memory processor 再调一次模型，总结这轮失败经验
3. 最后 summary 节点再调一次模型

如果 30 轮都失败，那么：

- proposer：约 `30` 次
- memory processor：约 `30` 次
- summary：约 `1` 次

合计就是 `61` 次。

这意味着这次 run 的请求数几乎已经被“proposer + memory + summary”完全占满，没有留下额外请求空间给工具回合。  
这与 `leansearch/search_count = 0` 完全一致。

结论：  
这次 `T1.free` 中，模型不仅没有“有效使用 LeanSearch”，而是连 tool call 都几乎没有发起。

## 7. 但搜索以后也不代表就能成功

probe 还有一个更细的结论：

- 它确实调用了 LeanSearch
- LeanSearch 返回了 `MulChar.sum_eq_zero_of_ne_one`
- 模型直接把这个结果抄成：

```lean
simpa using MulChar.sum_eq_zero_of_ne_one (R' := ℂ) hχ
```

但这依然失败了。

查看 mathlib 源码可知，`MulChar.sum_eq_zero_of_ne_one` 确实存在于：

- `Mathlib.NumberTheory.MulChar.Basic`

也就是说它不是一个假的 theorem 名。  
问题在于：

1. 当前文件没有引入对应模块；
2. 当前目标里的 `Character G := G →* ℂˣ`，而搜索结果的 theorem 是关于 `MulChar R R'` 的；
3. 目标还涉及把 `ℂˣ` coercion 到 `ℂ` 后求和，不能直接照抄 theorem 名。

所以即使“强制搜索”能把 `search_count` 从 0 变成正数，它也只能说明：

- free 模式下“不搜索”是一个真实问题

但仍然还留着另一个问题：

- 模型不会验证搜索结果是否真的适用当前文件环境与类型

## 8. 对这次 run 的最终判断

综合证据，我的判断是：

### 8.1 不是 LeanSearch 服务异常

因为：

- 直接调用工具能返回结果
- 低成本 probe 中模型在强提示下也能正常调用 LeanSearch

### 8.2 主要问题是 free 模式下没有足够强的工具使用诱导

当前 ATP 按你的要求，free 模式不给额外 prompt。  
而 ax-prover 的 iterative system prompt 对“应优先使用工具”这件事提示不够强，所以 `gpt-5.3-codex` 在 `T1.free` 上默认走了：

```text
猜 theorem 名 -> 编译报 unknown identifier -> 再猜 theorem 名 / 改几行 API -> 继续失败
```

### 8.3 次要问题是搜索结果采用阶段缺少“本地适用性校验”

即使让它搜了，它仍然会：

- 搜到一个接近的 theorem
- 不验证 import
- 不验证 theorem 类型是否和当前目标兼容
- 直接照抄进 proof

所以搜索不是充分条件，只是必要条件之一。

## 9. 对后续实验的意义

这次 run 给出的最重要结论是：

1. `T1.free` 当前失败不能再简单归因为环境路径问题  
   这次已经是真正进入 proof loop 后的 agent 行为问题。

2. 如果继续坚持“free 模式完全不加任何额外 prompt”，那么当前 ax-prover 默认 iterative prompt 下，`T1.free` 很可能长期都不会主动使用 LeanSearch。

3. 如果想让 free 模式仍然保持“不给路线提示”，但又不至于完全不搜索，那么最小的中间方案应当是：
   - 只加入一条中性的元规则  
   - 例如“当 theorem 名不确定时，先使用 `search_lean`，不要直接猜名字”
   - 这不属于 route hint，只属于工具使用纪律

4. 即使如此，仍需进一步解决“搜索结果适用性校验”问题。否则会从“不搜索”进化成“搜索后照抄错 theorem”。
