## 6. `user_comments` 和 theorem prompt 的区别

这里需要把两个概念分开：

### 6.1 `theorem_catalog.yaml` 里的 `free_instruction / disable_instruction / route_a.instruction`

这是 ATP 自己按场景生成 prompt 的来源。  
它们本身不会自动进入 ax-prover，必须经过：

- [render_user_comments()](/Users/hdm/math/elementary-number-theory/ATP/src/atp_axbench/prompts.py)

再写入 ax-prover 的：

- `prover.user_comments`

也就是说：

- YAML 里的 theorem prompt 是“场景级提示源”
- 是否真正注入，取决于 `render_user_comments(...)`

### 6.2 `ax_prover_experiment.yaml` 里的 `prover.user_comments`

这是 ax-prover 原生配置项，用来承载最终注入给 ax-prover 的附加提示。

但在 ATP CLI 里，它并不是直接照着 YAML 用的。  
[build_ax_config()](/Users/hdm/math/elementary-number-theory/ATP/src/atp_axbench/runner.py) 会在最后强行叠一层：

```python
{"prover": {"user_comments": user_comments}}
```

这里的 `user_comments` 来自 `render_user_comments(scenario)`。

因此，在当前 ATP CLI 运行链路里：

- `ax_prover_experiment.yaml` 中写 `user_comments: null`
- 实际上只是一个占位默认值
- 运行时会被 ATP 重新覆盖

这也是为什么把它设成 `null` 没问题：

- 这样表示“默认不在 ax YAML 里写死任何全局提示”
- 最终由 ATP 按场景决定要不要注入

## 7. 是否应该直接改 `ax_prover_experiment.yaml` 里的 `user_comments`

### 7.1 结论

**在当前 ATP CLI 链路下，单纯修改 `ax_prover_experiment.yaml` 里的 `user_comments`，基本不会解决你现在的问题。**

原因是它会被 ATP 运行时覆盖掉。

### 7.2 真正有作用的修改位置

如果你想让 `free` 模式也带上一条“先搜后猜”的中性规则，真正应当改的是：

- [prompts.py](/Users/hdm/math/elementary-number-theory/ATP/src/atp_axbench/prompts.py)

也就是把 free 模式从：

```python
return ""
```

改成：

- 返回一个非常短的、不中断 free 性质的元规则
- 例如：  
  “若 theorem 名不确定，先调用 `search_lean` 验证，不要直接猜 theorem 名”

### 7.3 会不会干扰 ax-prover

如果你把这条规则写成“路线提示”或“证明策略提示”，会干扰。  
但如果只写成一条非常中性的工具纪律，风险相对小：

- 不指定 Route A / Route B
- 不告诉它用哪种数学方法
- 只限制“不要先猜 theorem 名”

这种改动更像是“工具使用规范”，不是“证明路线 prompt”。
